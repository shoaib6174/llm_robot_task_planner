## Work Done

### Teleport Bounce Fix
The old implementation teleported the grasped object every 0.5s inline during
`move_to()`. With subprocess latency of 0.3-0.8s, this created visible bouncing
where the cube fell under gravity between teleports.

Fix: Replace inline loop with a background thread that runs continuously while an
object is grasped.

Key changes to `arm_controller_node.py`:
- Added `_teleport_thread` and `_teleport_stop` (threading.Event) to `__init__`
- Added `_start_teleport_thread()`, `_stop_teleport_thread()`, `_teleport_loop()`
- Thread starts after `self.grasped_object` is set in `execute_pick`
- Thread stops before release in `execute_place` (and manual gripper open)
- `move_to()` simplified to just `send_pose + time.sleep(settle)` — no inline teleport
- Removed per-teleport log spam (thread fires continuously)
- Thread uses `Event.wait(timeout=0.05)` between iterations to prevent busy-spin

The thread runs during navigation too — keeping the cube attached to the gripper
for the entire duration (46+ seconds in Scenario 1).

Scenario 3 limitations: After multiple rotation scans (detect_object calls), AMCL
localization degrades. The particle filter doesn't recover well from continuous
rotation in symmetric rooms. This is a known limitation (D030, S07) and would need
AMCL parameter tuning or alternative localization for multi-step tasks.
