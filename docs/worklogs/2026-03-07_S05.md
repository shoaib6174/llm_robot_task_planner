### TF Time Sync Fix (Key Fix)
- **Problem**: Bridging Gazebo's `/tf` topic (Pose_V type) brought ALL model poses into ROS TF, conflicting with robot_state_publisher's joint TFs. This caused hundreds of "moved backwards in time" warnings and stopped navigation mid-path.
- **Solution**: Removed `/tf` bridge entirely. Created `odom_to_tf_node.py` — a lightweight node that subscribes to `/odom` (nav_msgs/Odometry) and broadcasts just the `odom→base_footprint` TF transform. This is the only TF from Gazebo that Nav2 needs; all other transforms come from robot_state_publisher.
- **Result**: Zero time sync errors. Clean navigation.

### Navigation Testing

### Return Navigation Fix (Multiple Issues)
1. **Global costmap obstacle_layer blocking doorway**: Lidar marks at doorway walls + inflation blocked planner. Fix: removed obstacle_layer from global costmap (static map has walls).
2. **DWB controller U-turn drift**: DWB couldn't handle 180° turns, robot drifted into walls. Fix: switched to Regulated Pure Pursuit (RPP) controller.
3. **Mecanum wheel collision drift**: Angled rollers in collision mesh caused lateral motion during rotation with DiffDrive. Fix: replaced wheel collision STL with simple cylinders.
4. **RPP collision detection vs local costmap**: RPP's collision check used local costmap (with obstacle_layer), immediately rejected paths through doorway. Fix: disabled RPP collision detection.
5. **Final config**: RPP with allow_reversing, A* planner, no collision detection, no obstacle_layer in global costmap.

### Navigation Results
- A→B→A round trips: 100% success rate (2/2 round trips)
- Zero time sync errors, zero recovery behaviors needed
- Navigation time: ~14-15s per leg
